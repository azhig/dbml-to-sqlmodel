# Patch Application in Code to DBML

## Обзор

Команда `code to dbml` теперь применяет изменения к DBML файлу как патч, вместо полной перезаписи файла. Это позволяет сохранить форматирование, комментарии и структуру исходного файла.

## Как это работает

1. **Генерация DBML** - Код из `output/models/` конвертируется в DBML формат (нормализованный)
2. **Сравнение** - Нормализованные версии сравниваются чтобы определить, есть ли изменения
3. **Diff** - Создается unified diff между **оригинальным** файлом и **сгенерированным**
4. **Применение патча** - Патч применяется к **оригинальному** файлу напрямую
5. **Запись** - Результат сохраняется, сохраняя структуру и порядок таблиц

## Пример

Предположим, у вас есть файл `schema.dbml`:

```dbml
Table products {
  id serial4 [pk]
  direction_id bigint [not null]
  name text [not null]
}
```

Вы изменили модель в коде:

```python
class Product(SQLModel, table=True):
    __tablename__ = "products"
    id: Optional[int] = Field(default=None, primary_key=True)
    direction_id2: int = Field(...)  # Переименовали поле
    name: str = Field(...)
```

Команда `code to dbml` создаст diff:

```diff
--- a/schema.dbml
+++ b/schema.dbml
@@ -1,5 +1,5 @@
 Table products {
   id serial4 [pk]
-  direction_id bigint [not null]
+  direction_id2 bigint [not null]
   name text [not null]
 }
```

И применит только это изменение к файлу, сохранив всё остальное форматирование.

## Технические детали

### Функция `apply_diff_to_file`

Реализована в `src/dbml_to_crud/utils.py`:

```python
def apply_diff_to_file(file_path: Path, original: str, modified: str) -> bool:
    """
    Apply diff between original and modified to the actual file.

    Args:
        file_path: Path to the file to patch
        original: Original content (normalized/compared version)
        modified: Modified content (new version)

    Returns:
        True if changes were applied, False if no changes
    """
```

### Ключевые особенности реализации

1. **Работа с переносами строк**: Функция работает со строками без `keepends`, затем собирает результат с `\n.join()`, что предотвращает дублирование переносов строк
2. **Сохранение trailing newline**: Если оригинальный файл заканчивался на `\n`, это сохраняется в результате
3. **Обработка в обратном порядке**: Hunks обрабатываются от конца к началу файла, чтобы изменения в начале не влияли на номера строк в конце

### Зависимости

Добавлена библиотека `unidiff>=0.7.5` для парсинга и применения unified diff патчей.

### Алгоритм

1. Генерируется unified diff между оригинальным и измененным содержимым
2. Diff парсится с помощью библиотеки `unidiff`
3. Для каждого hunk (блока изменений):
   - Определяется позиция в файле (номер строки)
   - Собираются контекстные строки и добавленные строки
   - Удаленные строки пропускаются
   - Заменяются соответствующие строки в файле
4. В случае ошибки применения патча, выполняется fallback на полную перезапись

### Обработка ошибок

Если патч не может быть применен (например, из-за конфликтов), система:
1. Выводит предупреждение в консоль
2. Делает fallback на полную перезапись файла
3. Сообщает пользователю о fallback

## Преимущества

- ✅ Применяет только фактические изменения (patch-based)
- ✅ **Сохраняет оригинальный порядок таблиц**
- ✅ **Сохраняет оригинальную структуру и комментарии**
- ✅ Более безопасно для версионного контроля
- ✅ Легче отслеживать изменения в git diff

## Важные замечания

**Ограничения patch-based подхода**:
- Патч работает с **конкретными номерами строк**
- Если ваш файл имеет **очень разное форматирование** от генерируемого (например, таблицы в другом порядке, другой синтаксис), патч может не применится корректно
- В этом случае сработает **fallback** - файл будет полностью перезаписан нормализованной версией
- После первой перезаписи последующие изменения будут применяться как патчи корректно

## Fallback на полную перезапись

В следующих случаях система использует полную перезапись файла:
- Файл не существует (создается новый)
- Ошибка при парсинге diff
- Ошибка при применении патча
- Конфликты в содержимом файла
